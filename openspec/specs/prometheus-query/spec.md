## ADDED Requirements

### Requirement: Prometheus metric discovery
The system SHALL fetch available metric names from Prometheus via the `/api/v1/label/__name__/values` endpoint and present them for selection in the query builder.

#### Scenario: List available metrics
- **WHEN** user opens the query builder for a Prometheus datasource
- **THEN** the system SHALL display a searchable list of available metric names

#### Scenario: Metric list with search filtering
- **WHEN** user types in the metric search field
- **THEN** the system SHALL filter the metric list to show only metrics matching the search text

### Requirement: Prometheus label discovery
The system SHALL fetch label names for a selected metric via `/api/v1/labels` (with `match[]` parameter) and label values via `/api/v1/label/<name>/values`.

#### Scenario: List labels for a metric
- **WHEN** user selects a metric in the query builder
- **THEN** the system SHALL display available label names for that metric (excluding `__name__`)

#### Scenario: List label values
- **WHEN** user selects a label name for filtering
- **THEN** the system SHALL display available values for that label, scoped to the selected metric

### Requirement: Prometheus guided query builder
The system SHALL provide a guided query builder that allows users to:
- Select a metric name
- Add label filters (label = value)
- Select an aggregate function (optional: sum, avg, min, max, count)

The guided builder SHALL generate valid PromQL from these selections.

#### Scenario: Simple metric query
- **WHEN** user selects metric `temperature_celsius` with no filters
- **THEN** the system SHALL generate PromQL: `temperature_celsius`

#### Scenario: Query with label filter
- **WHEN** user selects metric `temperature_celsius` and adds filter `location="kitchen"`
- **THEN** the system SHALL generate PromQL: `temperature_celsius{location="kitchen"}`

#### Scenario: Query with aggregation
- **WHEN** user selects metric `temperature_celsius` with aggregate function `avg`
- **THEN** the system SHALL generate PromQL: `avg(temperature_celsius)`

### Requirement: Prometheus raw PromQL query mode
The system SHALL support a raw PromQL editing mode where users can write arbitrary PromQL expressions, using the existing `isRawQuery` and `rawQuery` fields on `SavedQuery`.

#### Scenario: Enter raw PromQL
- **WHEN** user switches to raw query mode and enters `rate(http_requests_total[5m])`
- **THEN** the system SHALL use the entered PromQL as-is for query execution

#### Scenario: Toggle between guided and raw mode
- **WHEN** user switches from guided mode to raw mode
- **THEN** the system SHALL pre-fill the raw editor with the PromQL generated by the guided builder

### Requirement: Prometheus range query execution
The system SHALL execute range queries via `/api/v1/query_range` with `query`, `start`, `end`, and `step` parameters derived from the selected time range.

#### Scenario: Execute range query
- **WHEN** a dashboard panel or widget requests data for a Prometheus query with time range "last 1 hour"
- **THEN** the system SHALL call `/api/v1/query_range` with appropriate start/end timestamps and auto-calculated step

#### Scenario: Auto-calculate step parameter
- **WHEN** time range is 1 hour
- **THEN** step SHALL be approximately 15 seconds
- **WHEN** time range is 24 hours
- **THEN** step SHALL be approximately 1 minute
- **WHEN** time range is 7 days
- **THEN** step SHALL be approximately 5 minutes

### Requirement: Prometheus result mapping to QueryResult
The system SHALL map Prometheus API responses to the app's `QueryResult` format:
- Matrix results: each sample becomes a row with `time` (ISO 8601), `value`, and one column per label
- Vector results: each element becomes a single row with `time`, `value`, and label columns
- Scalar results: a single row with `time` and `value`

#### Scenario: Map matrix result
- **WHEN** Prometheus returns a matrix result with multiple time series
- **THEN** each data point SHALL become a `QueryResult.Row` with time, value, and label columns
- **AND** series SHALL be distinguishable by their label column values

#### Scenario: Map vector result
- **WHEN** Prometheus returns an instant vector result
- **THEN** each element SHALL become a single row with the latest timestamp, value, and labels

### Requirement: DataSourceServiceProtocol conformance
`PrometheusService` SHALL conform to `DataSourceServiceProtocol` with the following mapping:
- `testConnection()` → Query `/api/v1/query?query=up` and verify success
- `query(_:)` → Execute the PromQL string as a range query using the default time range
- `fetchMeasurements()` → Return metric names via `/api/v1/label/__name__/values`
- `fetchFieldKeys(measurement:)` → Return `["value"]` (Prometheus metrics are single-valued)
- `fetchTagKeys(measurement:)` → Return label names for the given metric via `/api/v1/labels?match[]=<metric>`
- `fetchTagValues(measurement:tag:)` → Return values for a label via `/api/v1/label/<tag>/values?match[]=<metric>`

#### Scenario: Fetch measurements returns metric names
- **WHEN** `fetchMeasurements()` is called
- **THEN** the system SHALL return a list of all metric names from Prometheus

#### Scenario: Fetch field keys returns value
- **WHEN** `fetchFieldKeys(measurement:)` is called for any metric
- **THEN** the system SHALL return `["value"]`

#### Scenario: Fetch tag keys returns label names
- **WHEN** `fetchTagKeys(measurement:)` is called with metric name `cpu_usage`
- **THEN** the system SHALL return label names like `["host", "cpu", "mode"]` (excluding `__name__`)

#### Scenario: Fetch tag values returns label values
- **WHEN** `fetchTagValues(measurement: "cpu_usage", tag: "host")` is called
- **THEN** the system SHALL return the distinct values for the `host` label on the `cpu_usage` metric
